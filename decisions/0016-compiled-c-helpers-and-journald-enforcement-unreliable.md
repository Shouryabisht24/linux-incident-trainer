# 0016 — Compiled C helpers for shell-inexpressible mechanisms; journald's own internal enforcement is not reliable to break in this environment

## Decision
1. Several challenges added in the Phase 4 follow-up pass (`perm-setuid-helper-bit-stripped`, `proc-memory-leak-runaway`, `proc-fd-leak-too-many-open-files`, `pkg-missing-shared-library-symlink`, `net-port-conflict-stale-process`) compile a small, purpose-built C program with `gcc` in `seed.sh` at build time, rather than expressing the break in shell. This is a new pattern (the first 27 challenges were all pure shell/config).
2. Two originally-planned "Logs & journald" challenge ideas were abandoned mid-build after live testing, and replaced: a per-service journald rate-limit challenge (`RateLimitIntervalSec`/`RateLimitBurst`), and a persistent-journal-directory-permissions challenge (breaking `/var/log/journal/<machine-id>`'s group/ACL so a `systemd-journal`-group member can't read it). **Do not build either of these again** without first re-confirming the underlying journald behavior actually reproduces in this Docker Desktop / nested-cgroup environment.

## Why
**Compiled helpers.** Some mechanisms genuinely cannot be expressed in a shell script:
- setuid is deliberately *not* honored by the kernel on anything with a `#!` shebang (a well-known hardening measure against a classic TOCTOU race) — a real setuid-privilege-escalation-helper challenge requires a real compiled binary.
- Precise, deterministic memory/fd growth (`proc-memory-leak-runaway`, `proc-fd-leak-too-many-open-files`) is awkward and slow to do accurately in `sh` (no real `malloc`/`open` primitives without spawning a process per call) — a tiny C loop is simpler and behaves exactly as intended.
- A locally-built, versioned shared library (`pkg-missing-shared-library-symlink`) needs to actually be an ELF `.so` with a real embedded `SONAME` for `ldconfig` to do anything realistic with it.
- A process that genuinely holds a TCP port open without answering (`net-port-conflict-stale-process`) needed a real `bind()`/`listen()` socket, not a shell workaround.

`gcc`/`libc6-dev` are installed via `apt-get` in the Dockerfile exactly like every other package these challenges already pull in (`nginx`, `sudo`, `openssh-server`, etc.) — this only touches image *build* time, which already has network access for `apt-get`; it does not violate the challenge container's own runtime network isolation (`decisions/0001`/AUTHORING.md's network rules are about the running session container, not the `docker build` step).

**journald enforcement.** Both abandoned ideas were built and run through the full verify loop before being dropped:
- The rate-limit challenge set `RateLimitIntervalSec=30`/`RateLimitBurst=3` via a `journald.conf.d` drop-in (confirmed correctly merged via `systemd-analyze cat-config systemd/journald.conf`), then burst-emitted messages via `logger`. All messages were delivered every time, burst after burst — no suppression was ever observed, and no "Suppressed N messages" note appeared in journald's own log either. journald's rate limiting groups messages by the sender's cgroup, and processes started via plain `docker exec` in this environment don't appear to get attributed to a distinguishable-enough cgroup/unit for the limiter to engage — this may be specific to Docker Desktop's cgroup driver/nesting, not a general Debian/systemd behavior.
- The directory-permissions challenge pre-created `/var/log/journal/<fixed-machine-id>` at build time with `root:root 0700` (intending a "hardening pass locked it down" break). At container boot, the directory's owner/group/mode/ACL were found to already be `root:systemd-journal`, `2755`-equivalent, with a `group:adm:r-x` ACL entry — i.e. **systemd's own tmpfiles/journald setup silently re-asserts the directory's permissions and ACL at boot**, overwriting whatever was baked into the image. The break never survived to the point the trainee would see it. (Separately, read access is actually granted via ACL to the `adm` group specifically, not via the base `systemd-journal` group-owner bits — a real, useful gotcha for a *future* attempt at this idea, but not why this one was dropped.)

Both were replaced with `logs-rsyslog-facility-blackholed` (an rsyslog `& stop` rule routing a facility to `/dev/null` before it reaches the real rule) — a mechanism entirely within our own authored config, not dependent on journald's internal enforcement, and it verified cleanly on the first attempt.

## How to apply
- Prefer a compiled C helper over shell contortions whenever a challenge needs setuid, precise resource-consumption behavior, or a real listening socket. Follow the existing pattern: write the `.c` to `/tmp` in `seed.sh`, `gcc -O2 -o <dest> <src>`, `rm -f` the source, install `gcc libc6-dev` in the Dockerfile.
- Don't design a challenge around journald's own internal rate-limiting or its automatic directory/ACL management without first proving the specific behavior survives a real `docker build && docker run` cycle in this project's environment — it may be silently neutralized (rate limiting) or silently healed (directory permissions) before the trainee ever interacts with it. journald *config* drop-ins themselves (`Storage=`, as in `logs-journald-not-persistent`) are fine and already proven to work; it's journald's own runtime enforcement/self-healing behavior that's unreliable here.
